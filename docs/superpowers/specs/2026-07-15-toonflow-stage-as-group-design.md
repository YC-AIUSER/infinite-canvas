# Toonflow 环节组化(垂直切片:资产库) · 设计文档

- 日期:2026-07-15
- 状态:已通过 brainstorming,待出实现计划
- 涉及:`web/src` 画布(Toonflow 生成流)
- 范围:**只做一个环节(资产库)的垂直切片**,验证收益与代价后再谈推广

## 背景与目标

### 问题的由来

Toonflow 生产画布(`buildToonflowCanvasTemplate`)把 14 个环节各做成**一个 320×190 的小节点**,一字排开连成流水线。每个环节实际产出的**集合**——资产卡、镜头、首帧、视频——全塞在该节点的 `metadata.toonflow.output.payload` 里,靠**弹窗(modal)**逐项查看/编辑。

用户指出:**每个环节的本质是一个"装着一堆子项的容器(组)",而无限画布原生就支持 Group + 子节点**,当前"小节点 + 弹窗"是在跟画布的天性对着干。

### 对照原版 Toonflow(用户要求核对)

原版 `Toonflow-web`(Vue3 + Vue Flow)也是"一环节一节点",但**节点内部把整个集合铺开成卡片网格**,每张卡可单独编辑/删除/重生成:

- `assets.vue`:`cardGrid`,每张资产卡铺在节点里,原卡→衍生卡嵌套网格。
- `storyboard.vue`:`frameGrid`,每个镜头一张卡。
- `workbench.vue`:唯一走"小节点 + 弹窗"(时间线剪辑器)。

**结论:原版每个环节节点就是一个"装着卡片的容器",集合明明白白铺在节点里、逐项可操作。我们移植版把它退化成了小节点 + 弹窗,是一次倒退。**

关键的是,原版数据模型是"环节持有一个数组,每个元素渲染成可操作卡片"——**卡片是数组的视图,不是独立图节点**。这实际上就是下文"镜子路线"的做法。

### 目标

用无限画布的原生 Group + 真实子节点,把原版本来就有、移植时丢掉的"集合铺开 + 逐项可操作"补回来,并更进一步(子项是画布一等节点,能自由拖动)。

**本期只做资产库一个环节**,作为验证切片。

## 关键决策(brainstorming 结论)

### 决策 1:走"镜子路线",不走"本体路线"

整套 Toonflow 正确性机器(级联顺序、上游版本快照 `upstreamVersions`、`stale` 失效传播、接缝检查、导出汇总)**全部只认 `节点.metadata.toonflow.output.payload`**,并假设"一环节 = 一节点 = 一份 output"。

- **镜子路线(采纳)**:环节节点仍是唯一真相源,`payload` 一字不动;额外**投影**出一个组 + 每个子项一个真实子节点,子节点是 `payload` 的派生视图。单项操作写回 `payload[i]`,组随之刷新。
  - 收益:级联/版本/stale/导出**一行都不用改**,下游照读 `payload`。风险低、可回退(删掉组,`payload` 仍在)。
  - 印证:**原版就是这么做的**(环节持有数组,卡片是视图)。镜子路线是忠实还原,不是妥协。
- **本体路线(不采纳)**:子节点变权威数据,环节退化为纯组,级联/导出/版本机器全要围绕"一组 N 子节点"重写。风险集中在 Toonflow 最易出正确性 bug 的部分(T9/M2 刚清完),难回退。**留待镜子路线验证成功后再评估。**

### 决策 2:切片选「资产库」,不选首帧

在"九宫格直出"方法论下,`buildKeyframesPrompt` 明写"以故事板页线稿为唯一构图锁,只上色不改构图,共 N 格……单图上色":

- **首帧 / 故事板页 / 视频**:每段是**一张合成图 / 一条合成视频**,N 个镜头烤进一张九宫格里,**节点层面没有可铺开的子项**。组化只会得到"一个组里躺一张大图",无意义。
- **资产库 `cards[]`**:角色/场景/道具/衍生卡,**每张一个独立 `storageKey`(独立图)**,是货真价实的集合,也是原版最标志性的 `cardGrid`。

因此**资产库是唯一合适的首个切片**。(音频 `audioKeys[]` 是次要候选,本期不碰。)

### 决策 3:复用现有 batch 子节点的持久化成法

画布已有"图片批量展开":一个批次根节点通过 `batchRootId` / `batchChildIds` 关联多个**真实持久化子节点**,整体拖动/删除联动。子节点组化沿用同构做法(真实持久化节点 + 互链 + 生命周期联动),不新造机制。

### 非目标(本期不做)

- 其它 13 个环节的组化。
- 本体路线(子节点变权威、独立版本、跨组连线如"资产卡→引用它的镜头")。
- 组内嵌套(原卡→衍生卡的子分组)——本期**扁平铺**,所有卡平级。
- 删除现有资产卡弹窗——本期**并存**,先加组。

## 数据模型

### 真相源(不变)

资产库节点保持现状:

```
资产库节点.metadata.toonflow = {
  kind: "assets",
  status,
  output.payload.cards: AssetCard[],   // ← 唯一真相源,不动
  ...
}
```

`AssetCard`(见 `schema.ts`)已有 `cardId` / `cardType` / `name` / `anchor` / `parentCardId?` / `storageKey?`,足够投影,**无需改 schema**。

### 投影层(新增)

派生一个组节点与 N 个子节点:

**组节点**(`CanvasNodeType.Group`):
```
{
  id: `assets-group-<assetsNodeId>`,
  type: Group,
  metadata: {
    projectionOf: { stageNodeId: <资产库节点id>, kind: "assets" },  // 新增标记字段
  }
}
```

**子节点**(每张卡一个,`CanvasNodeType.Image`):
```
{
  id: `assets-card-<cardId>`,
  type: Image,
  metadata: {
    groupId: <组节点id>,                      // 复用现有归组机制
    storageKey: <card.storageKey>,            // 复用现有图片渲染
    cardProjection: { stageNodeId, cardId },  // 新增:回指真相源
  }
}
```

**关键不变量:子节点/组节点都不带 `toonflow.kind`** → `graphNodes()`(`node-runtime.ts`)只收有 toonflow 的节点,投影层**自动被级联/版本/导出/接缝忽略**。这是镜子路线零污染的技术保证,需有测试守卫。

### 新增 metadata 字段

在 `CanvasNodeMetadata`(`types/canvas.ts`)加两个可选字段:
- `projectionOf?: { stageNodeId: string; kind: ToonflowNodeKind }` —— 标记组是某环节的投影。
- `cardProjection?: { stageNodeId: string; cardId: string }` —— 标记子节点回指哪张卡。

## 同步行为(投影层 ⇄ 真相源)

所有同步只发生在投影层,**不调用任何级联/版本函数**。

| 事件 | 行为 |
|---|---|
| 资产卡生成/编辑成功(`applyAssetCardsSave` 之后) | 按 `cards[]` 重建组的子节点集合:新增卡→建子节点,删除卡→删子节点,已存在→更新 `storageKey`。子节点位置:已有的保位,新增的走现有防重叠摆位。 |
| 画布加载(`hydrateToonflowProject` 之后) | 若资产库有 `cards[]` 但无对应组,重建投影。 |
| 用户拖动子节点 | 仅更新该子节点 `position`(真实持久化,保位)。不影响 `cards[]`。 |
| 用户删除某子节点 | 从 `cards[]` 删除对应卡(走 `applyAssetCardsSave`),再重建投影。 |
| 用户对某子节点点"重生成" | 只重生成该卡(复用资产卡单卡生成路径),写回 `cards[i]`,再刷新该子节点。 |
| 资产库节点被删除 | 连带删除其组与所有子节点(仿 batch 根删除联动)。 |

**冲突原则**:`cards[]` 永远是真相。任何投影层与 `cards[]` 不一致时,以 `cards[]` 重建投影(投影可丢弃重算,真相不可)。

## 涉及文件(预估)

- `web/src/types/canvas.ts` —— 加 `projectionOf` / `cardProjection` 字段。
- `web/src/lib/canvas/`(新建投影模块,如 `toonflow-group-projection.ts`)—— 纯函数:`cards[]` → { 组节点, 子节点[] } 的构建与 diff/重建。**核心逻辑,可独立单测。**
- `web/src/pages/canvas/project.tsx` —— 在资产卡保存/加载/删除/单卡重生成处调用投影重建;子节点交互接线。
- `web/src/components/canvas/canvas-node.tsx` —— 子节点若需"重生成/删除"角标(可复用现有 hover toolbar)。
- 测试:投影层纯函数单测 + "子节点不进 toonflow 图"的守卫测试。

## 成功标准(验收看证据,非报告)

1. 生成资产后,画布出现「资产库」组,**组内每张卡一个真实图节点,铺开可见,不用开弹窗**。
2. 单卡子节点可选中、拖动;整组可拖动;删一张卡 → 对应子节点消失且 `cards[]` 同步。
3. 单卡重生成 → 子节点图更新、`cards[]` 版本正确。
4. **下游首帧/视频生成不受影响**(仍读 `payload.cards`),浏览器端到端跑通一段。
5. 现有全部测试仍绿(当前 198)+ 新增投影层单测与"零污染"守卫测试绿。
6. 刷新页面后组与子节点正确恢复,不产生脏节点/孤儿。

## 风险与缓解

| 风险 | 缓解 |
|---|---|
| 投影与 `cards[]` 不一致(双表征) | 单向:`cards[]` 是唯一真相,投影可整体重算;不做反向权威。 |
| 子节点误入级联/版本图 | 不带 `toonflow.kind`;加守卫测试断言 `graphNodes` 不含投影节点。 |
| 节点数量增长影响性能 | 资产卡量级有限(角色+场景+道具+衍生,通常几~十几张);仍复用已修的渲染记忆化。观察即可,本切片不预优化。 |
| 撤销/重做把投影与真相拆散 | 投影重建作为纯派生,尽量由真相变更后统一触发,避免把投影单独塞进撤销栈;实现计划阶段细化。 |

## 待实现计划阶段细化的开放问题

1. 单卡"重生成"是否已有现成路径可复用,还是需新增(影响成功标准 3 的工作量)。
2. 子节点位置持久化 vs 每次派生自动摆位的取舍(保位体验 vs 存储/撤销复杂度)。
3. 组标题/角标 UI 细节(是否显示环节名、卡片数、状态)。
